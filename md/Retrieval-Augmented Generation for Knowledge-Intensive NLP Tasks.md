# Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks

## 基本信息
- 论文：Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks
- 会议：NeurIPS 2020
- 链接：https://proceedings.neurips.cc/paper/2020/file/6b493230205f780e1bc26945df7481e5-Paper.pdf
- 关键词：RAG、Dense Retrieval、Parametric Memory、Non-parametric Memory、Seq2Seq

## 快速省流总结
这篇论文基本可以看作现代 RAG 范式的经典起点。作者认为，仅靠大模型参数内部记忆去回答知识密集型问题有三个明显短板：知识难更新、难以提供证据来源、容易幻觉。因此他们提出把“参数化记忆”和“外部非参数记忆”结合起来：外部知识库存放在 Wikipedia 的稠密向量索引里，先用神经检索器找相关文档，再让 seq2seq 生成器基于“问题 + 检索到的文档”生成答案。论文最核心的贡献不是“检索+生成”这件事本身，而是把它做成了一个**可端到端微调的统一概率模型**，并提出了两种代表性变体：**RAG-Sequence** 和 **RAG-Token**。前者假设一整个输出都依赖同一组检索文档，后者允许不同 token 在生成时依赖不同文档，因此更灵活，但推理更复杂。

## Motivation
作者的出发点很明确：预训练语言模型虽然在参数里“存了很多知识”，但它们并不擅长对知识进行可控访问和精确操纵。对于开放域问答、事实生成、知识密集型文本生成这类任务，单靠参数记忆会遇到三个问题：

1. **知识更新困难**：模型参数一旦训练完成，就不是很容易局部刷新知识。
2. **证据不可追溯**：模型生成答案时通常不能明确告诉你“它依据了哪段证据”。
3. **容易幻觉**：模型会生成听起来合理但缺乏依据的内容。

作者注意到，以前已经有一些“参数记忆 + 外部记忆”的方法，但大多集中在抽取式任务，或者需要为某个特定任务从头设计架构。他们希望找到一种更通用的方式：既继承大规模预训练 seq2seq 模型的生成能力，又能借助外部知识库增强事实性和可更新性。

## Challenges
这篇论文要解决的问题其实不只是“先检索再生成”这么简单，而是更细的三个层面：

### 1. 如何让检索和生成真正耦合起来
如果只是先独立检索，再把 top-k 文档简单拼接给生成模型，这种方式虽然有效，但检索器本身未必会为了最终生成效果而优化。作者想做的是一个统一模型，让检索器和生成器可以一起为了下游任务目标进行训练。

### 2. 如何在生成过程中处理“多个检索文档”
检索出来的文档通常不止一篇。问题在于，答案究竟依赖哪篇文档？是整个答案都依赖同一篇，还是不同部分可以依赖不同证据？这直接决定了模型结构与推理方式。

### 3. 如何在可解释性、效率与生成质量之间做平衡
如果允许每个 token 都动态切换文档，表达能力会更强，但推理和训练都会更复杂。反过来，如果整句甚至整段都只绑定一篇文档，又可能过于僵硬。作者因此提出两个粒度不同的变体进行比较。

## Overview / 核心思路
RAG 的整体结构非常清晰，可以拆成三层：

1. **非参数记忆（External Memory）**  
   使用整个 Wikipedia 构建稠密向量索引，作为可检索知识库。

2. **检索器（Retriever）**  
   使用 DPR（Dense Passage Retriever）根据输入问题到向量索引中找 top-k 文档。

3. **生成器（Generator）**  
   使用 BART 这类预训练 seq2seq 模型，在输入问题和检索结果的条件下生成最终输出。

作者把检索到的文档当作**潜变量**。也就是说，模型不是武断地认为某篇文档一定正确，而是对多个候选文档上的生成概率进行边缘化近似：  
- 先检索 top-k 文档  
- 再对这些文档对应的生成分布加权求和  
- 最终得到输出概率

这一步很关键，因为它把“多文档证据融合”变成了概率模型的一部分，而不是纯工程拼接。

## Technical Details / 方法细节（详细）
这部分是整篇论文最值得细读的地方。

### 1. 基本组成
RAG 由两部分组成：

- **Retriever**：DPR，输入为 query，输出为文档分布 \(p_\eta(z|x)\)
- **Generator**：BART，输入为 \(x\) 和检索文档 \(z\)，输出为生成概率 \(p_\theta(y|x,z)\)

其中：
- \(x\)：输入（如问题）
- \(z\)：检索文档
- \(y\)：目标输出序列

作者把 top-k 检索文档视为对潜变量 \(z\) 的近似支持集，然后在这些文档上做边缘化。

### 2. RAG-Sequence
RAG-Sequence 的假设是：**整个输出序列都依赖同一份检索文档**。  
也就是说，一旦某篇文档被选中，整句答案都基于它来生成。

其直觉可以理解成：
- 先从 top-k 文档里软选择一篇（或几篇带权重的文档）
- 再在该文档条件下生成完整答案
- 最后对不同文档对应的生成概率求和

这种做法的优点是：
- 概率建模简单
- 推理更稳定
- 对短答案问答任务很合适

但缺点也明显：
- 如果答案的不同部分需要依赖不同文档，RAG-Sequence 表达力不够

### 3. RAG-Token
RAG-Token 更进一步，假设：**每一个输出 token 都可以依赖不同的检索文档**。  
也就是说，在解码过程中，模型可以在 token 级别重新决定该参考哪份证据。

直观理解：
- 第一个词可能依赖文档 A
- 第二个词可能依赖文档 B
- 后续 token 还能继续切换

这使它比 RAG-Sequence 更灵活，特别适合多事实拼接、复杂生成或需要跨文档融合的情况。  
但这带来了两个代价：
- 解码更复杂
- 计算开销更高

所以这两者本质上是两个不同粒度的“潜变量边缘化方案”。

### 4. 检索器与生成器如何连接
在实现上，retriever 先把 query 编码成向量，然后从 Wikipedia 向量库中做最大内积检索，拿到 top-k passages。生成器收到的输入不是单纯问题，而是类似：

`[question ; retrieved passage]`

每一篇 passage 都会和原问题配对送入生成器，生成器在这些候选证据条件下计算输出概率，再由 retrieval probability 对这些概率进行加权。

因此，RAG 的核心不是“把检索结果拼接进 prompt”，而是：
- 检索器给出一个**文档分布**
- 生成器给出一个**条件生成分布**
- 最终对 latent documents 做**概率边缘化**

### 5. 为什么可以端到端训练
这是这篇论文最经典的一点。作者没有把 retriever 固定死，而是让下游生成损失反向影响检索器。  
直观上：
- 如果某些检索文档更有助于生成正确答案
- 那么训练会推动 retriever 提高这些文档的概率
- 从而形成“为生成服务的检索”

这就让 RAG 不只是两个模块串起来，而是一个统一优化框架。

### 6. 外部知识库的意义
外部知识库是 non-parametric memory。它的最大价值在于：
- 可以替换
- 可以增量更新
- 可以追溯来源
- 不需要重新训练整个大模型

这也是后来几乎所有工程化 RAG 系统都会继承的思想：模型负责推理与表达，知识库存放外部世界事实。

## Implementation & Experiment
作者在多个知识密集型任务上进行了实验，包括：
- 开放域问答
- 问题生成
- 摘要类生成等

整体上，RAG 在开放域 QA 上取得了很强的结果，并且相比纯参数化模型：
- 更 factual
- 更 specific
- 更 diverse

另外，作者的人类评估也显示，RAG 生成的文本在事实性和信息量上优于纯 BART 基线。

## 结果怎么看
这篇论文最重要的实验结论不是“性能涨了多少”，而是它证明了三件事：

1. **检索增强生成在知识密集任务上有效**
2. **把检索文档建模为潜变量并做边缘化是合理的**
3. **生成模型不必把所有知识都死记到参数里，外部知识库可以成为长期可维护的记忆层**

## 优点
### 1. 奠定了现代 RAG 的基本范式
“外部知识库 + 稠密检索 + 生成模型”这条主线，后面几乎所有工作都在此基础上演化。

### 2. 提供了可解释性
答案可追溯到检索到的 passage，而不是完全黑箱地由参数生成。

### 3. 支持知识更新
更新知识库通常比重新训练模型容易得多。

### 4. 同时适用于问答与生成
不是只针对 extractive QA，而是面向 seq2seq generation 的统一方案。

## 局限性
### 1. 检索只发生在输入阶段
这意味着它更适合“问题一开始就能确定信息需求”的任务。对于长文本生成、逐步展开推理、多跳检索，它还不够灵活。

### 2. 文档粒度仍偏粗
如果真正有用的信息只在局部段落、句子或图结构片段里，top-k 文档级别检索可能仍会引入噪声。

### 3. 生成器对检索噪声较敏感
一旦检索结果偏了，生成器可能被误导。

### 4. 缺少“何时检索”的判断
它默认每次都要检索，这一点后来被 Self-RAG、Adaptive RAG、DRAGIN 之类工作继续推进。

## 和后续工作的关系
这篇论文可以看作后面很多方向的起点：
- **SELF-RAG**：在“是否检索、检索后如何自我批判”上继续发展
- **DRAGIN / Active Retrieval / FLARE**：把检索时机从输入前一次扩展为生成过程中的动态触发
- **Graph RAG / Repo-level RAG**：把知识源从平面文本扩展到图和代码仓库结构
- **Corrective / Ranking RAG**：开始处理“检索错了怎么办”与“多个上下文怎么排优先级”

## 对你当前研究的启发
如果映射到你现在的 VRagent / XR 测试计划生成系统，这篇论文最重要的启发有四点：

1. **把 scene/script/object 索引看作 non-parametric memory**
   你的 scene graph、层级结构、脚本摘要，本质上都可以看作外部知识库，而不是让 LLM 硬背。

2. **retrieval 不应该只是 prompt 拼接，而应该服务最终 plan 生成**
   也就是说，你后面可以进一步考虑让检索模块依据 verifier / execution feedback 共同优化。

3. **多证据融合很关键**
   你不一定只要一个对象说明块，可能还需要脚本逻辑、依赖关系、历史失败案例共同支持一个 action plan。

4. **RAG-Sequence vs RAG-Token 的思想可以迁移到 planning 粒度**
   例如：
   - 一个完整 plan 只基于同一批证据
   - 或者每一步 action 都可以依赖不同证据
   这就是典型的“sequence-level planning”与“step-level evidence grounding”的差别。

## 一句话总结
这篇论文的真正价值，不只是提出了一个叫 RAG 的名字，而是第一次把“外部检索知识 + 生成模型”系统化为一个统一、可微调、可解释、可扩展的生成框架，后面几乎所有 RAG 变体都能在这里找到原型。
